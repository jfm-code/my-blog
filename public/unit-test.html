Unit Testing

Things to know (Prof didn‚Äôt go through this üôÇ)

  • The reason we need unit testing is mainly to save time, to have something that test our code automatically
  • The ‚ÄúBoost Unit Test Framework‚Äù is what we will used for C++
  • To run the test, type make first into the terminal, then ./test
  • Structure of the test file:
    • #include <boost/test/unit_test.hpp> ‚áí include this most of the time to use the boost framework
    • #define BOOST_TEST_DYN_LINK ‚áí this means we are linking to the boost framework, and it will autocreates a main function for us, we don‚Äôt have to write one in that test file
    • #define BOOST_TEST_MODULE [name] ‚áí this syntax follows with a name that we custom based on what the tests are for, it is used to define a group of test cases (also called test module)
      • Example: #include BOOST_TEST_MODULE Uniquify Tests ‚áí the goal might be to uniquify something.

        ‚áí After this, we will need to include a header file, to store function declarations of the functions we want to test. In this case it might be called #include ‚Äúunique.hpp‚Äù

      • Example: #define BOOST_TEST_MODULE MathTest ‚áí the goal might be to test something related to maths

        ‚áí After this, we will need to include a header file, to store function declarations of the function we want to test. In this case it might be called #include ‚Äúmymath.hpp‚Äù

    • Make a unit test (test function) by using BOOST_AUTO_TEST_CASE that has multiple boost comparisons in it:
      BOOST_AUTO_TEST_CASE(TestAdd) //We can replace the "TestAdd" with any name we want
{
	BOOST_CHECK(add(2, 2) == 4);
	BOOST_REQUIRE(...);
	//OTHER BOOST COMPARISONS
}

Comparisons

  • BOOST_CHECK ‚áí verifies that some condition is true
  • BOOST_CHECK_EQUAL ‚áí verifies that the two parameters are the same.
    • Attention that if they are different, you can see what the actual values are, not just that it failed ‚áí much better to debug
  • BOOST_CHECK_GE ‚áí greater than or equal
  • BOOST_CHECK_GT ‚áí greater than
  • BOOST_CHECK_LE ‚áí less than or equal
  • BOOST_CHECK_LT ‚áí less than
  • BOOST_CHECK_NE ‚áí not equal
  • BOOST_CHECK_CLOSE(left, right, tolerance) ‚áí for floating-point numbers, left and right are close to each other within a specified tolerance
  • BOOST_CHECK_EQUAL_COLLECTIONS(first_begin, first_end, second_begin, second_end) ‚áí check if 2 arrays, or vectors are equal

‚áí Attention: We also see BOOST_REQUIRE_EQUAL, or BOOST_REQUIRE_EQUAL_COLLECTIONS, these are to verify and check things too, but with more restriction than BOOST_CHECK

  • Difference between BOOST_CHECK and BOOST_REQUIRE:
    • [test]_CHECK keeps going on a failure
      ‚áí Use when you can still get more information from further checks

      ‚áí This is mostly used when cases are independent, if one fails then others can still be fine

    • [test]_REQUIRE stops the test on a failure

      ‚áí End up throwing exceptions after

      ‚áí This is mostly used when one case fails may lead to all other cases fail or the whole program abort

  • Throw exceptions:
    • BOOST_REQUIRE_THROW ‚áí test if it does throw, the test fails if the code doesn‚Äôt throw exception
    • BOOST_REQUIRE_NO_THROW ‚áí the test fails if the code throws exception

Things to avoid when testing

  • Don‚Äôt test implementation details
    ‚áí We care about the input and output, what is done, not how it is done
  • Think about edge cases when testing, such as boundary values or negative values

Example 1

  • We‚Äôll have some tests to test our 2 functions, add() and sub()
//Math.hpp - where function declarations belong to
#pragma once
int add(int n1, int n2);
int sub(int n1, int n2);

//Math.cpp - where function implementations belong to
#include "Math.hpp"

int add(int n1, int n2)
{
	return n1+n2;
}

int sub(int n1, int n2)
{
	return n1-n2;
}
//test.cpp - where the test cases located
#define BOOST_TEST_DYN_LINK
#define BOOST_TEST_MODULE MathTest
#include <boost/test/unit_test.hpp>
#include "Math.hpp"

using namespace std;

BOOST_AUTO_TEST_CASE(TestAdd) //used to test add() function only
{
	BOOST_CHECK_EQUAL(add(2,2),4);
	BOOST_CHECK_EQUAL(add(-1,3),2);
	BOOST_CHECK_EQUAL(add(5,-5),0);
}

BOOST_AUTO_TEST_CASE(TestSub) //used to test sub() function only
{
	BOOST_CHECK_EQUAL(sub(2,2),0);
	BOOST_CHECK_EQUAL(sub(-1,3),-4);
	BOOST_CHECK_EQUAL(sub(5,-5),10);
}

BOOST_AUTO_TEST_CASE(TestVectorIndex) //used to test if throw exception or not
{
	vector<int>v = {0,1,2,3,4}; //v.size() = 4
	v.reserve(10); //v.capacity() = 10;
	for(int i=0;i<v.size();i++) //case that does not be out of bounds
	{
		BOOST_REQUIRE_NO_THROW(v.at(i)); //will pass this test because won't throw
		BOOST_REQUIRE_NO_THROW(v[i]);    //will pass this test because won't throw
	}
	for(int i=v.size();i<v.capacity();i++) //case that will go out of bounds
	{
		BOOST_REQUIRE_NO_THROW(v[i]);   //pass because won't check for boundary when using brackets
		BOOST_REQUIRE_THROW(v.at(i), out_of_range); //std::out_of_range	
	}
}

Example 2

  • Explain things that appear in the code:
    • uniquify()
      • We see this function in our code, where does it come from?

        ‚áí his function is not in STL of C++, it is in the #include BOOST_TEST_MODULE Uniquify Tests , and Uniquify Tests is the test module that we created by ourself, so we will have to implement this uniquify() function ‚áí is used to removes duplicate elements from an array, vector, or list, leaving only unique elements.

      • In other words, this function is what will be tested by our tests. Yes, we use tests to test this uniquify() function written by us
      • The function declaration of this function will be located at ‚Äúunique.hpp‚Äù
    • Why are there so many files?
      • At first there are only files in the src folder. After we type make into the terminal, files in the bin are generated (those are output files).
      • Once we have output file, we can now use test to check (cause we only care about input and output while checking), so if we dont have outputs, how do we check?
    • Explanation of what‚Äôs in each file
      • test.cpp ‚áí where our test cases against versions of uniquify() function are written
      • unique.hpp ‚áí function declaration of uniquify()
      • unique1.cpp / unique2.cpp / ‚Ä¶ ‚áí where different versions of uniquify() implementation are written

      ‚áí In short, we are writing test cases to test against 9 versions of uniquify()

//test.cpp for uniquify()
#define BOOST_TEST_DYN_LINK
#define BOOST_TEST_MODULE Uniquify Tests
#include <boost/test/unit_test.hpp>
#include "unique.hpp"

using namespace std;

BOOST_AUTO_TEST_CASE(Test1) {
    vector<int> v = {};
    uniquify(v); //call function to remove duplicates of the vector
    BOOST_REQUIRE_EQUAL(v.size(),0); //check if size of v is 0 or not

    vector<int> expected = {};
    BOOST_REQUIRE_EQUAL_COLLECTIONS(v.begin(),v.end(),expected.begin(), expected.end());
}

BOOST_AUTO_TEST_CASE(Test2) //this test passes the unique4 version of function
{ 
    vector<int> v = {1,2,1};
    uniquify(v);
    //after uniquify, the resulting v can be {1,2} or {2,1}, it doesn't have to be the first number 1 removed
    //so we shouldn't make an expected vector like {1,2} because that won't cover the case {2,1}
    //one thing we know for sure is that there will be only one number 1 left, and one number 2 left after uniquify
    
    BOOST_REQUIRE_EQUAL(count(v.begin(),v.end(),1), 1); //count(v.begin(),v.end(),1) will count the occurence of 1 in vector v
    BOOST_REQUIRE_EQUAL(count(v.begin(),v.end(),2), 1);
}

BOOST_AUTO_TEST_CASE(Test3) //this test passes the unique9 version of function
{
    vector<int> v = {1,2,3,2,2,3};
    uniquify(v);
    BOOST_REQUIRE_EQUAL(v.size(),3);
    BOOST_REQUIRE_EQUAL(count(v.begin(),v.end(),1), 1);
    BOOST_REQUIRE_EQUAL(count(v.begin(),v.end(),2), 1);
    BOOST_REQUIRE_EQUAL(count(v.begin(),v.end(),3), 1);
}
